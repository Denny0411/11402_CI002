## Thinking Logic and Solution Strategy

> ### 題目故事與核心概念
> 
> 一般我們在寫程式時，佇列 (Queues) 和優先佇列 (Priority Queues) 是大家都很熟悉的資料結構 。但這道題目要我們模擬的是在日常生活中常見，卻比較少被提出來討論的「團隊佇列 (Team Queue)」 。你可以把它想像成午餐時間在學生餐廳排隊買飯的狀況 。
> 
> 在團隊佇列中，有兩個最重要的規則：
> 
> * **團隊歸屬**：每個在排隊的元素，都隸屬於某一個特定的團隊 。
> 
> * **排隊規則 (Enqueue)**：當一個元素要進入佇列時，它會從頭到尾檢查佇列中是不是已經有「同隊的隊友」在排隊了 。如果有，它就會直接安插在隊友們的最後面 。如果沒有任何隊友在排隊，它就只能乖乖排在整個佇列的最尾端 。
> 
> * **離開規則 (Dequeue)**：取出元素的方式跟一般的佇列一模一樣，就是從最前面的頭部開始，依序處理並讓元素離開佇列 。
> 我們的任務，就是寫出一個程式來完美模擬這個團隊佇列的運作 。
> 
> ### 輸入格式說明
> 
> * **測試案例**：輸入檔中會包含一個或多個測試案例 。
> 
> * **團隊數量**：每個測試案例一開始會有一個數字 t ($1 \le t \le 1000$)，代表團隊的總數 。
> 
> * **團隊成員**：接下來會有 t 行的團隊描述，每一行會先給出該團隊有幾個元素，接著列出屬於該團隊的所有元素編號 。
> 
> * **元素範圍**：元素的編號是介於 0 到 999999 之間的整數，且一個團隊最多可以有 1000 個元素 。
> 
> * **操作指令**：接下來會出現一系列的指令，總共有以下三種 ：
> 
>   * **ENQUEUE**：把指定的元素加入團隊佇列中 。
> 
>   * **DEQUEUE**：處理並移除目前排在佇列最前面的元素 。
> 
>   * **STOP**：代表這個測試案例的指令結束 。
> 
> * **結束條件**：當輸入的團隊數量 t 為 0 時，代表所有的輸入結束，程式應該停止執行 。
> 
> ### 輸出格式說明
> 
> * **情境編號**：針對每個測試案例，必須先印出一行 `Scenario #k`，其中的 k 代表是第幾個測試案例 。
> 
> * **取出結果**：對於輸入中的每一個 `DEQUEUE` 指令，你必須在單獨的一行印出被取出的元素編號 。
> 
> * **格式細節**：每個測試案例輸出完畢後，都必須印出一個空白行，包含最後一個測試案例也要印 。
>

> ### 開發思路與效能警告 (重點教學)
> 
> 題目特別給了一個**警告 (Warning)**：一個測試案例可能會包含高達 200000（二十萬）個指令，因此我們實作的團隊佇列必須非常有效率 。不管是加入 (Enqueue) 還是取出 (Dequeue) 元素，都必須在常數時間，也就是 $O(1)$ 的時間內完成 。
> 
> 這意味著，**我們絕對不能在每次有新元素要加入時，真的從頭到尾去掃描陣列尋找隊友**，這樣時間複雜度會太高，導致超時 (Time Limit Exceeded)。
> 
> **建議的實作步驟與邏輯：**
> 
> 1. **紀錄元素的團隊歸屬**：我們需要一個能快速查詢「某個元素屬於哪一隊」的機制。因為元素編號最大只到 999999，你可以考慮使用一個大陣列（或 Hash Map）來記錄每個元素對應的團隊編號。
> 2. **準備一個「主佇列 (Main Queue)」**：這個佇列用來記錄目前「有哪些團隊」正在排隊，以及他們排隊的先後順序。
> 3. **準備多個「團隊專屬佇列 (Team Queues)」**：你可以建立一個佇列陣列，總共有 t 個佇列。每個佇列專門用來存放該團隊目前正在排隊的成員。
> 4. **當執行 ENQUEUE 時**：先查出這個元素屬於哪一隊。接著檢查該隊的「團隊專屬佇列」是不是空的。如果原本是空的，代表這隊沒有人在排隊，你就必須把這隊的編號加入「主佇列」的尾端；最後，把這個元素加入該隊的「團隊專屬佇列」中。
> 5. **當執行 DEQUEUE 時**：先看一下「主佇列」最前面是哪一個團隊。接著去那個團隊的「團隊專屬佇列」中，把最前面的成員拿出來並印出。如果拿出來之後，發現該隊的「團隊專屬佇列」空了，那就代表這隊的人都處理完了，必須把這個團隊的編號從「主佇列」中移除。
>

### 所有的成員都儲存到同一個 `team_id` 陣列嗎？

> 在你的 `afterClass_v1.cpp` 檔案中，我們可以把原本的 `while (t--)` 區塊改寫成一個 `for` 迴圈，這樣會更容易追蹤現在正在處理第幾個團隊。
> 
> 開發邏輯如下：
> 
> 1. 使用一個外層迴圈來跑每個團隊（例如變數 `i` 從 0 到 `t-1`，代表團隊編號）。
> 2. 在迴圈內，先讀取這個團隊「有幾個人」。
> 3. 接著寫一個內層迴圈，根據人數讀取每一個「成員編號 (`person`)」。
> 4. 將讀到的成員編號當作 Index，把當前的團隊編號 `i` 存進去，也就是寫下 `team_id[person] = i;`。
